# SHUD Agent Harness 建设方案（汇报稿）

## 一、项目一句话定义

建设一套**面向 SHUD / rSHUD / AutoSHUD 持续优化的科研 Agent 系统**：  
以大模型为“指挥官型科研助手”，以可控的工程环境为“harness（运行支撑层）”，把**科研问题—实验设计—模型运行—证据分析—代码优化—再验证**连成一个闭环。

---

## 二、为什么现在值得做

### 1. SHUD 已经不是单一模型，而是一个“模型系统”
目前我们面对的不是一个孤立程序，而是至少三层联动体系：

- **SHUD**：数值求解与物理过程核心
- **rSHUD**：数据预处理、输入生成、结果分析与可视化
- **AutoSHUD**：自动建模、流程编排、批量运行

随着模型复杂度上升，团队遇到的问题不再只是“某段代码有 bug”，而是：

- 科学问题如何快速转化为可执行实验
- 不同仓库之间如何协同演化而不互相破坏
- 模型优化是否真的有科学增益，而不是偶然提高某个案例分数
- 老经验、老脚本、老 benchmark 如何沉淀为团队资产，而不是只存在个人脑子里

### 2. 传统研发方式的瓶颈越来越明显
如果继续依赖“人工读代码 + 手工跑实验 + 手工写总结”的模式，会出现几个问题：

- **效率瓶颈**：问题提出后，实验设计、运行、复盘周期长
- **可重复性不足**：同一结论换一个人不一定复现得出来
- **跨仓库改动成本高**：改 SHUD 常常同时牵涉 rSHUD 与 AutoSHUD
- **知识沉淀弱**：一次调试经验、一次失败教训，难以系统复用

### 3. Agent 的价值，不是替代科研，而是放大科研带宽
本项目不是让 AI 替代学者做科学判断，而是让 AI 承担高频、重复、耗时的科研工程工作，例如：

- 基于问题自动形成实验草案
- 自动跑小流域 / 小案例 benchmark
- 自动诊断运行失败和回归退化
- 自动形成对比报告和证据包
- 在受控条件下完成跨仓库代码修改与验证

最终目标是：

> **让科研人员把更多时间放在“提出问题、判断证据、决定方向”上，  
> 把更多体力活交给 Agent 系统完成。**

---

## 三、这套系统到底要建什么

我们要建设的不是一个“聊天机器人”，也不是一个简单的 CI/CD 脚本集合，而是一套：

> **Commander Agent + Harness Runtime + SHUD Model Stack**

可以理解为三层：

### A. 指挥层：Commander Agent
这是系统主控。

它负责：

- 理解科研问题
- 判断下一步是做实验、做诊断、改代码，还是先请求人工审批
- 调用执行型 Worker
- 汇总结果，更新长期记忆
- 根据失败与成功经验逐步改进自己的工作方式

### B. 支撑层：Harness Runtime
这是让 Agent 真正可靠工作的“运行支撑层”。

它负责：

- 提供受控的 bash / 代码 / 文件系统环境
- 提供 benchmark、日志、报告、验证器
- 记录每次行为与结果
- 管理技能（skills）和记忆（memory）
- 控制权限边界，必要时强制人工审批

### C. 模型层：SHUD / rSHUD / AutoSHUD
这是被优化的对象。

系统会围绕三类真实任务工作：

1. **科研任务**  
   例如：为什么半干旱流域暴雨事件中洪峰偏低？

2. **工程任务**  
   例如：如何把 event-scale diagnostics 加到 SHUD 输出，并同步更新 rSHUD 和 AutoSHUD？

3. **验证任务**  
   例如：这次改动是否真的提高模型质量？是否破坏旧案例？是否引入兼容性问题？

## 四、我们采用的核心思路

## 1. 主控是 LLM，但不能裸奔
系统主控应是一个**指挥官型 Agent（Commander Agent）**，而不是写死的流程脚本。  
但这个 Commander 不是无限自由的，它必须遵守一套“科研宪法”：

- 不能没有问题意识就改代码
- 不能没有 baseline 就宣称模型变好
- 不能绕过水量平衡、数值稳定性和旧案例回归
- 高风险动作必须走人工审批

也就是说：

> **主控由 LLM 负责决策，流程由 harness 负责约束。**

## 2. 工具层以 bash 为主，而不是堆砌很多 API
不采用过于复杂的工具层设计。  
系统的主要动作空间就是：

- `bash`
- `read`
- `write`
- `glob`
- `grep`
- `delegate_worker`
- `read_memory`
- `propose_memory_update`
- `request_human_gate`

这意味着 Agent 可以像研究工程师一样，在受控沙箱中：

- 看代码
- 写脚本
- 生成脚手架
- 编译 SHUD
- 跑 rSHUD / AutoSHUD
- 读日志
- 修问题
- 产出报告

这样更贴近真实科研软件工作的形态，也更利于系统持续进化。

## 3. skills 与 memory 是能力增长的核心
系统不是靠一次 prompt 变强，而是靠长期积累变强。

- **skills**：可复用的程序化能力模块  
  例如：`run-shud-tiny-case`、`diagnose-shud-run-failure`、`benchmark-before-after`

- **memory**：长期认知状态  
  包括失败经验、领域知识、已有主张、历史改动、证据与反证

随着系统使用次数增加，它会越来越熟悉：

- 哪些案例最容易暴露问题
- 哪些代码修改风险最高
- 哪些诊断逻辑最有效
- 哪些实验设计最有区分度

---

## 五、系统的最小结构

为了避免过重设计，我们把系统收敛成 **4 个核心角色**：

### 1. Commander Agent
长期运行的主控，负责整体策略与任务分派。

### 2. Worker Agent
短期执行者，负责某一段具体任务，例如：

- 跑实验
- 写脚本
- 改代码
- 修测试

### 3. Critic Agent
审查者，不主控、不直接改代码，专门负责：

- 反证
- 审查是否验证充分
- 识别是否过拟合某个案例
- 判断是否需要补 benchmark / 补实验

### 4. Harness Optimizer
元层优化器，用于改进系统自身，而不是直接改 SHUD 模型。  
它负责分析：

- 哪些 skill 经常失效
- 哪些 context 组织方式不合理
- 哪些失败模式在重复发生
- 如何让 Commander 与 Worker 更可靠

## 六、这套系统将为 SHUD 团队带来什么

### 1. 让“问题到实验”的速度显著提升
今天一个科研问题从提出到形成一轮可执行实验，往往需要较长准备时间。  
引入 Agent 后，很多环节可以自动化完成：

- 草拟实验设计
- 整理输入数据
- 生成脚本
- 批量运行
- 汇总结果

### 2. 让“代码优化”不再脱离科学证据
传统代码修改容易出现“工程上改了很多，科学上未必更好”。  
这套系统要求：

> 每次重要代码变更，都必须绑定实验、证据和验证报告。

### 3. 让跨仓库联动更可靠
SHUD、rSHUD、AutoSHUD 之间存在天然耦合。  
Agent 系统可以把跨仓库修改纳入统一闭环：

- 发现接口影响
- 同步修改
- 自动验证
- 输出变更说明

### 4. 让团队经验成为“可执行知识”
过去很多经验停留在个人层面，例如：

- 某类 solver failure 怎么排查
- 某类 benchmark 最容易暴露问题
- 哪些输出格式变更最危险

这套系统会把这些经验逐步沉淀成：

- 可调用的 skills
- 可检索的记忆
- 可复用的脚手架和模板

### 5. 让实验记录、代码记录、决策记录统一起来
最终形成的不只是“代码仓库”，而是一个更完整的科研记录体系：

- 问题为什么提出
- 实验怎么设计
- 改动为什么发生
- 结论有什么证据
- 为什么最终接受或拒绝某次改动

---

## 七、项目边界：做什么，不做什么

### 本项目要做的
- 围绕 SHUD / rSHUD / AutoSHUD 建设 agent runtime
- 建立 benchmark 与验证闭环
- 建立 skill / memory / report 体系
- 让 Agent 能在受控环境中辅助科研与代码优化
- 建立人机协同的审批机制

### 本项目不做的
- 不追求“全自动科研”
- 不让 Agent 独立决定物理机制是否成立
- 不在第一阶段就追求大而全的平台化
- 不依赖 GitHub Issue 作为科研入口
- 不优先做 UI 外壳，而是先把核心能力做扎实

---

## 八、分阶段建设路线

## 阶段 1：最小闭环（1–2 个月）
目标：先跑通一个“问题—实验—报告—小改动—验证”的闭环。

交付物：

- Commander Agent 原型
- bash 沙箱
- tiny benchmark
- 一套基础报告模板
- 5 个以内核心 skills
- 受控 memory 写入机制

这一阶段的目标不是“很强”，而是“跑通、可验证、能复盘”。

## 阶段 2：跨仓库工程闭环（2–4 个月）
目标：让系统真正能协同 SHUD / rSHUD / AutoSHUD。

交付物：

- 跨仓库 worktree / patch / validation 流程
- event diagnostics、round-trip tests、benchmark before/after
- Critic Agent 审查回路
- 证据包与变更说明自动化

## 阶段 3：自我进化与长期任务（4–6 个月）
目标：让系统不只是执行任务，还能改进自己。

交付物：

- Harness Optimizer
- skill promotion / deprecation 机制
- 失败模式归纳与自动修复建议
- 多案例、多流域 benchmark 策略
- 长时段科研任务跟踪与交接

---

## 九、资源需求建议

建议以“小而强”的方式启动：

### 人员配置（建议）
- **1 名科研负责人**：把控科学问题与关键判断
- **1 名研究软件工程负责人**：负责系统实现、Agent runtime、工具链
- **0.5–1 名模型/数据支持**：维护 benchmark、数据与脚本

### 计算与软件资源
- 一台可稳定运行本地/服务器任务的开发机
- HPC 或服务器队列资源（用于批量实验）
- 代码仓库与实验存储空间
- 适量模型 API / 本地模型推理预算

## 十、风险与控制

### 风险 1：Agent 可能“看起来会做，实际乱做”
控制方式：

- 沙箱隔离
- 强制 benchmark
- Critic 审查
- 高风险动作人工审批

### 风险 2：为了自动化而过度复杂
控制方式：

- 坚持 bash-first
- 工具极简
- 先做最小闭环，再逐步演化

### 风险 3：代码优化脱离科学问题
控制方式：

- 每次变更绑定 Research Change Set
- 每次重要结论都要求证据包
- 不允许只凭单案例分数提高宣称“模型变好”

### 风险 4：系统知识被错误写入长期记忆
控制方式：

- memory proposal → critic 审查 → commander 确认 → 才能升级为长期记忆

---

## 十一、项目成功的衡量标准

项目成功不看“Agent 会不会聊天”，而看下面这些指标：

### 科研效率指标
- 从问题提出到得到一轮可执行实验的时间是否显著缩短
- 从运行失败到定位原因的时间是否显著缩短

### 工程质量指标
- 重要变更是否都有验证报告
- 跨仓库修改是否更少引入兼容性问题
- benchmark 回归检测是否更及时

### 知识沉淀指标
- 是否形成可复用 skill 库
- 是否形成长期可检索记忆
- 是否减少“同类错误反复出现”

### 人机协同指标
- 人类是否把更多时间投入到科学判断，而不是重复性操作
- 人类 review 是否从“逐行体力审查”转向“关键决策审查”

---

## 十二、建议的立项结论

建议将该项目定位为：

> **SHUD 团队的下一代科研软件基础设施项目**

它不是一个单独的小工具，而是未来几年内支持 SHUD 持续演化的重要底座。  
如果建设成功，它不仅能提升当前模型研发效率，还会沉淀出一套对实验科学软件普遍有价值的方法论。

---

## 十三、建议老板拍板的三件事

1. **同意按“小步快跑”的方式立项**  
   先做最小闭环，不一开始追求大平台。

2. **明确项目的第一目标是“可用、可控、可复盘”**  
   而不是“全自动”“大而全”。

3. **将该项目视为科研能力建设，而不只是软件开发**  
   因为它本质上是在增强团队处理复杂科研软件系统的能力。

---

## 附：一句最直白的话

如果把 SHUD 研发看成一支科研队伍，那么：

- 学者负责提出问题、做关键判断；
- Agent 负责跑腿、做实验、查日志、改代码、写初稿；
- Harness 负责给 Agent 提供纪律、工具、记忆和边界。

这套系统建成后，团队最宝贵的资源——**人的时间与注意力**——会被更集中地用在真正高价值的科学判断上。
