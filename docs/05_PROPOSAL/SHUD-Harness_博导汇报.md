# 关于建设 SHUD-Harness 科研工程平台的汇报

> [!summary] 一句话结论
> 拟建设 **SHUD-Harness**：一套让 SHUD 模型研发**每一步都有证据、每一个科学结论都经过 PI 审批**的科研工程平台。平台不替代研究者做科学判断，而是把运行、验证、报告和复盘组织成标准流程，让 PI 和课题组把时间花在提出问题和判断证据上，而不是花在重复性的工程劳动上。

![[assets/final-workbench-effect.png]]
_图 1：SHUD-Harness 工作台效果图。研究者在浏览器中创建任务、跟踪运行、查看图表、审阅报告、作出科学决策。_

## 1. 我们面临什么问题

SHUD 模型系统是一套完善的复杂生态：

| 仓库 | 作用 | 当前痛点 |
|------|------|----------|
| **SHUD** | C++ 水文数值求解核心 | 物理方程、参数、输出格式改动影响全链路，但没有系统性验证 |
| **rSHUD** | R 工具包：输入输出、分析、可视化 | 与 SHUD 输出格式强耦合，一端改了另一端容易断 |
| **AutoSHUD** | R 自动建模流水线 | 脚本链长、依赖多，换一台机器经常跑不通 |
| **Zero** | Agent/Web 运行时基础设施 | 需要改造成带科研证据约束的 runtime |

**核心矛盾**：模型系统越来越复杂，但研发方式还停留在"手工跑命令 + 口头记录 + 零散脚本"的阶段。

具体表现为四类高频问题：

**问题一：跑通了，但说不清为什么。**
一次敏感性分析涉及哪个版本的代码、哪份数据、什么参数配置，事后回溯全靠个人记忆。换个人、换台机器，结果可能不一样。

**问题二：改了一处，不知道会断哪里。**
给 SHUD 加一个新的诊断输出，需要同步检查 rSHUD 的 reader、AutoSHUD 的后处理脚本、已有的 benchmark 是否被影响。目前这些检查全靠人工，漏测是常态。

**问题三：指标改善了，但不确定是不是对的。**
参数校准让某个流域的洪峰误差下降了，但这个改善可能来自模型机制的改进，也可能来自补偿效应——一个公式写错了，刚好被另一个参数的偏差抵消。**没有系统性的方式区分校准改善和理论正确性。**

**问题四：经验散落在个人手里。**
上一个学生积累的调参经验、失败诊断记录、benchmark 结果，换个人就接不上。课题组缺乏结构化的科研资产沉淀。

![[assets/why-now-pressure.png]]
_图 2：SHUD 模型栈进入跨仓库演进阶段后面临的四类核心压力。_

> [!warning] 如果不解决这些问题
> 模型越复杂，重复劳动越多，可信度越难保证。长期拖慢科研产出的不是单次运行时间，而是运行前后的**组织成本、复盘成本和可信度成本**。

## 2. SHUD-Harness 是什么

**一句话定义**：SHUD-Harness 是一个 **Web 科研工作台**，它把 SHUD 模型的运行、验证、分析和报告组织成有证据链的标准流程。

**它不是什么**：
- 不是聊天机器人——它是有结构化流程和治理规则的科研平台
- 不是自治 AI——所有科学结论必须由 PI 审批
- 不是脚本集合——它提供从问题定义到报告交付的完整闭环

**它的工作方式**：

```
PI 在浏览器中提出研究问题
    ↓
平台把问题拆解为可执行任务
    ↓
自动编排模型运行、数据解析、图表生成
    ↓
生成结构化证据报告（不是一句"模型改善了"，而是完整的证据包）
    ↓
PI 审阅证据、留下批注、作出科学判断
    ↓
所有过程沉淀为可复盘的科研资产
```

### 2.1 平台由四层组成

| 层 | 面向谁 | 做什么 |
|----|--------|--------|
| **Web 工作台** | PI / 研究者 | 浏览器中创建任务、跟踪运行、查看图表、审阅报告、审批高风险操作 |
| **Agent 协调层** | 自动执行 | 把研究问题拆解为任务，编排模型运行，生成分析和报告草稿 |
| **证据治理层** | 保障可信度 | 记录每次运行的版本、数据、环境；生成证据报告；强制 PI 审批高风险操作 |
| **科学模型栈** | SHUD 体系 | SHUD / rSHUD / AutoSHUD 继续作为科学核心，平台只是它们的编排和验证外壳 |

![[assets/system-architecture.png]]
_图 3：SHUD-Harness 的平台边界——PI 负责科学判断，Agent 负责执行闭环，Harness 负责证据链与复现。_

## 3. 平台有哪些核心能力

平台能力分三个层面：**辅助研究**（帮你更快做出好研究）、**工程执行**（帮你省掉重复劳动）、**科研治理**（帮你保证结论可信）。

### 第一层：辅助开展研究

这是平台最直接的价值——**让研究者把精力集中在科学问题本身，而不是环境配置、手工运行和结果整理上。**

#### 能力一：从研究问题到可执行实验

PI 提出一个模糊的研究方向，平台帮你落地：

| PI 说 | 平台做 |
|-------|--------|
| "ccw 洪峰偏低，看看是参数还是结构问题" | 拆解为：先做 ksat/roughness 敏感性 → 再做结构诊断 → 生成对比方案 |
| "对比一下 heihe 和 ccw 的蒸散发响应" | 确定两个流域的数据/版本 → 设计对比实验 → 定义评价指标 |
| "新加的湖泊模块对已有结果有没有影响" | 识别影响范围 → 设计 regression 测试 → 跑 before/after 对比 |

Agent 生成的不是最终方案——它是**供 PI 审阅、修改和批准的草案**。PI 可以调整参数空间、更换评价指标、添加约束条件，直到方案符合研究意图。

#### 能力二：跨仓库代码理解

SHUD 生态有四个仓库、数万行代码，研究者换一个课题方向就需要重新理清代码关系。平台内置 **Repo Explorer**，可以：

- 快速定位某个物理过程（比如入渗）在 SHUD 源码中的实现位置
- 追踪一个输出变量从 SHUD 写出 → rSHUD 读入 → AutoSHUD 使用的完整链路
- 分析一处代码改动会影响哪些下游函数和输出
- 为新加入课题组的研究生提供**结构化的代码导览**，而不是让他们从零读代码

这不是代替研究者理解代码，而是把"在四个仓库里翻找半天"变成"几秒钟拿到结构化上下文"。

![[assets/capability-research-plan-repo-explorer.png]]
_图 4：从 PI 研究问题到 AnalysisPlan 草案，Repo Explorer 同步给出跨仓库代码上下文和影响面。_

#### 能力三：智能诊断与失败分析

模型运行失败或结果异常时，不再只看一堆日志猜原因：

- **运行失败**：Agent 分析退出码、日志关键词、资源状态，给出可能原因和建议动作
- **数值异常**：检查水量平衡残差、负存储、CVODE 收敛失败等数值健康指标
- **结果偏离**：与基线对比，标注哪些变量、哪个时段、哪些网格出现显著偏差
- **参数搜索停滞**：连续多组参数没有改善时，建议转换策略（从调参转向检查输入数据、从 roughness 转向 soil/landcover mapping）

诊断结论记录在运行记录中，下次遇到类似问题可以直接查阅，不再依赖个人记忆。

#### 能力四：经验沉淀与知识复用

课题组积累的科研经验不再随人员更替流失：

- **调参经验**：哪些参数对哪类流域/事件敏感，什么范围合理，记录为结构化的 MemoryNote
- **失败教训**：某种配置导致了什么问题，怎么解决的，保留为可检索的诊断记录
- **Benchmark 基线**：每个流域/事件的基线运行、指标和环境快照，长期维护
- **可复用技能**：常用分析流程（比如"ccw 暴雨事件标准诊断"）可以封装成 Skill，一键触发
- **PI 批注**：PI 在报告上的科学判断、研究方向指导、假设取舍，沉淀为课题组的决策记录

新研究生加入课题组时，可以查阅：这个流域之前做过什么实验、哪些路径被验证、哪些被否定、PI 当时怎么判断的。**从"师兄口述传承"变成"平台记录可查"。**

![[assets/capability-diagnostics-memory.png]]
_图 5：运行失败后，工作台把日志、数值健康检查、历史 MemoryNote 和推荐动作放在同一个诊断界面中。_

### 第二层：工程执行

#### 能力五：一键运行，全程留痕

研究者不再需要手敲命令和记录环境。平台会自动：
- **锁定版本**：记录运行时 SHUD / rSHUD / AutoSHUD 的代码版本、编译环境和依赖版本
- **追踪数据**：记录输入数据来源、预处理参数、观测数据引用
- **执行运行**：在受控沙箱中运行 SHUD，实时推送日志和进度
- **管理长任务**：SHUD 运行可能几小时，平台提交任务后释放资源，结果回来自动恢复（Park/Resume）
- **保存证据**：命令、日志、输出、指标、图表全部归档，随时可追溯

#### 能力六：跨仓库改动验证

修改 SHUD 输出格式？平台会自动检查：
- 旧输出格式是否被破坏
- rSHUD reader 是否需要更新
- AutoSHUD 后处理是否受影响
- 小 benchmark（ccw tiny）是否仍然通过

所有检查结果打包成 **兼容性证据**，PI 审批后才能合入。

#### 能力七：敏感性分析与参数校准

支持批量参数扫描、敏感性分析和校准实验：
- 定义参数空间和评价指标
- 自动运行参数组合，实时显示每组状态（排队/运行中/成功/失败）
- 生成敏感性热力图、过程线对比、误差指标表
- **必须有基线对比**：任何"改善"声明都要说清楚跟谁比、条件是否一致

![[assets/capability-execution-sensitivity.png]]
_图 6：批量运行、参数表、过程线对比、敏感性热力图和下一步建议在同一工作台中联动。_

### 第三层：科研治理

#### 能力八：结构化证据报告

平台生成的不是一句"模型改善了"，而是一份完整的 **EvidenceReport**：

1. 研究问题是什么
2. 使用了什么数据、代码和环境版本
3. 运行了什么命令，产生了什么日志
4. 指标和图表
5. 当前证据支持什么、**不支持什么**
6. 需要 PI 判断的问题
7. 建议的下一步

报告有内置**语言约束**：
- ✅ 允许写："在 ccw 这个事件窗口内，ksat 扰动对洪峰误差有明显影响"
- ❌ 禁止写："模型机制已经被验证"或"该方案普遍有效"
- ❌ 禁止写："校准结果证明物理假设正确"

#### 能力九：PI 审批与高风险管控

所有高风险操作必须经过 PI 审批：

| 操作 | 是否需要 PI 审批 |
|------|:---:|
| 运行参数扫描 | 否（低风险） |
| 修改物理方程 | **是** |
| 修改模型假设 | **是** |
| 修改默认参数的物理意义 | **是** |
| 修改输出变量含义 | **是** |
| 替换 benchmark 基线 | **是** |
| 删除原始数据 | **禁止或 PI 审批** |

PI 的审批不是简单的"同意/拒绝"——可以留下科学批注，这些批注会成为课题组的长期科研记录。

![[assets/pi-evidence-review.png]]
_图 7：PI 与研究工程师围绕 EvidenceReport 审阅结果，而不是围绕零散日志做判断。_

## 4. 科学正确性如何保障

这是 SHUD-Harness 与一般自动化工具的根本区别。

### 4.1 核心原则：搜索是下游工具，不是科学正确性的替代品

机器学习项目可以用 trial-and-error：改模型、跑实验、看指标。但 SHUD 是物理模型，**校准能掩盖公式错误**。举个具体例子：

> 如果入渗方程的符号写反了（水往上走而不是往下），但恰好通过调高另一个参数来补偿，校准指标可能反而更好。这类错误在参数搜索中不会被发现，只有回到公式层面才能识别。

因此，SHUD-Harness 的治理原则是：

```
先确保公式正确，再谈参数搜索。
```

### 4.2 理论到代码证据链

对于涉及物理方程、模型假设、数值方案的改动，平台要求建立完整的证据链：

```
理论说明（为什么要改、改了什么假设）
    ↓
公式记录（符号、单位、量纲、边界条件）
    ↓
推导审查（推导步骤、使用了什么假设、有无未决问题）
    ↓
数值方案（离散化类型、守恒性预期、稳定性风险）
    ↓
代码映射（公式的哪一项对应代码的哪个函数/变量）
    ↓
验证用例（不是校准，而是证明"代码忠实表达了公式"）
    ↓
证据报告 → PI 审批
    ↓
审批通过后，才允许进入参数搜索和校准
```

这条链路不是可选的装饰——**对于高风险科学变更，每一步都是必须的**。

### 4.3 三个概念必须分开

| 概念 | 回答的问题 | 谁能判断 |
|------|-----------|---------|
| **Verification**（验证） | 代码实现是否忠实表达了公式？ | 平台可提供证据，Reviewer/PI 判断 |
| **Validation**（确认） | 模型能否解释观测和现实过程？ | 只有 PI 能判断 |
| **Calibration**（校准） | 给定模型结构，哪组参数让指标最好？ | 平台可计算，但**不等于科学结论** |

平台会在报告中严格区分这三者。校准改善 ≠ 理论验证，这是 Agent 绝对不允许混淆的底线。

### 4.4 Agent 行为边界

| Agent 可以做 | Agent 不可以做 |
|-------------|--------------|
| 拆解任务、编排运行 | 替代 PI 做科学判断 |
| 生成报告草稿 | 把自己的推导标记为"已接受" |
| 指出指标变化 | 把校准改善写成"理论已验证" |
| 提供验证证据 | 把验证通过写成"模型结构普遍真实" |
| 建议下一步 | 跳过 PI 审批直接执行高风险变更 |

## 5. 一次典型研究是什么样的

### 场景一：参数敏感性分析（低风险）

PI 提出：
> "ccw 暴雨事件洪峰偏低，先做 ksat 和 roughness 的敏感性分析，不要修改物理方程。"

平台执行：

1. **创建任务**：记录研究目标（洪峰偏低）和约束（禁止改方程）
2. **锁定环境**：记录 SHUD / rSHUD / AutoSHUD 版本和 ccw 数据
3. **制定计划**：Agent 生成参数扫描方案，PI 审阅并调整参数空间和指标
4. **运行基线**：先跑一次当前参数的基线，作为对比基准
5. **批量扫描**：自动运行参数组合，PI 在工作台看每组进度
6. **生成图表**：过程线对比、敏感性热力图、误差指标表
7. **生成报告**：EvidenceReport 列出证据和局限性
8. **PI 决策**：继续做 holdout 验证 / 尝试工程改动 / 暂缓

全程 PI 可以随时介入：调整参数空间、排除失败实验、对报告加批注。

![[assets/scenario-low-risk-sensitivity.png]]
_图 8：低风险敏感性分析场景中，PI 可以在批量运行过程中修改参数范围、排除失败实验并选择下一步。_

### 场景二：修改物理方程（高风险）

PI 提出：
> "当前入渗与地下水的耦合方式可能不准确，考虑修改耦合项。"

平台执行（比场景一多了科学审查环节）：

1. **创建任务**，系统自动识别为高风险（涉及物理方程）
2. **理论说明**：Agent 整理科学问题、涉及的物理过程、假设和非目标
3. **公式记录**：列出相关方程、符号、单位、量纲检查
4. **推导审查**：记录推导步骤和使用的假设 → **PI 审阅推导**
5. **数值方案**：说明离散化变化、守恒性预期、稳定性风险
6. **代码映射**：标注修改了 SHUD 的哪个文件/函数、对应公式的哪一项
7. **验证用例**：运行水量平衡检查、极限条件测试、regression 对比
8. **证据报告** → **PI 审批**
9. 审批通过后，**才允许**进入参数敏感性或校准

![[assets/scenario-high-risk-physics-gate.png]]
_图 9：高风险物理方程修改场景中，Theory-to-Code 证据链和 PI gate 会阻止系统直接进入参数搜索。_

> [!important] 关键区别
> 场景一（参数搜索）可以直接执行；场景二（修改方程）必须先走完理论审查，PI 同意后才能搜索。这就是"搜索后置于科学正确性"的具体体现。

## 6. 整体建设方案

### 6.1 六阶段渐进建设

平台不是一步建完，而是分六个阶段逐步成型。每个阶段只激活该阶段需要的能力，避免一开始就把所有功能都压上去。

| 阶段 | 建设重点 | 交付的能力 |
|------|----------|-----------|
| **Phase 1** 骨架可运行 | monorepo 基座、共享 schema、task API、Web 壳 | 浏览器可创建任务、刷新可恢复、最小质量门禁 |
| **Phase 2** 执行闭环 | RunJob 提交、沙箱执行、日志流、Park/Resume | 模型可以在平台中运行，长任务可暂停恢复 |
| **Phase 3** 科学运行 + 理论治理 | ccw tiny 实跑、**Theory-to-Code 证据链**、VerificationCase、科学变更 gate | 公式→代码可追踪，高风险变更需 PI 审批 |
| **Phase 4** 分析引擎 + 搜索 gate | 敏感性/校准/benchmark、**搜索前置检查**、实验账本 | 批量分析、基线对比、搜索必须在理论审查后 |
| **Phase 5** 报告与治理 | EvidenceReport、**语言约束**、PI gate、HTML 导出 | 结构化报告、禁止校准写成验证、审批留痕 |
| **Phase 6** 集成交付 | 多用户、CI/CD、部署、运维 | 小团队可部署使用的完整平台 |

### 6.2 六个始终不变的底线

无论建到哪个阶段，以下底线不动摇：

1. **PI 主导**：Agent 执行，PI 判断。科学结论不交给 AI。
2. **证据优先**：没有版本记录、数据溯源和运行记录的结论，不进报告。
3. **搜索后置**：参数搜索/校准是下游工具，不能替代理论审查。
4. **8 个核心对象**：TaskCard、StackLock、DataProvenance、RunJob、RunRecord、AnalysisPlan、EvidenceReport、ChangeRequest——不膨胀。
5. **只读源码**：SHUD / rSHUD / AutoSHUD 原始仓库只读，所有改动在隔离工作区中完成。
6. **失败证据保留**：失败的运行、被拒绝的修改都保留证据，不允许删除。

## 7. 系统架构概览

技术上采用 TypeScript 全栈（Bun + Hono + React），在 Zero Agent Runtime 基础上扩展 SHUD 领域逻辑。

![[assets/system-architecture-detailed.png]]
_图 11：系统架构图。控制流从 PI 进入 Web 工作台，证据流从 SHUD 运行经 RunRecord / Artifact 回到 EvidenceReport 和 PI Decision。_

平台内部有两条必须分开的流：

| 流 | 方向 | 作用 |
|----|------|------|
| **控制流** | PI → 工作台 → Agent → 模型执行 | 研究者提出任务、审阅计划、跟踪运行、审批报告 |
| **证据流** | 模型运行 → 运行记录 → 证据报告 → PI 决策 | 把命令、日志、指标、图表、决策沉淀为可复盘证据 |

关键设计：Web 工作台只承载实时展示，不作为证据源。报告只能引用持久化的运行记录和 artifact。

### 与四个仓库的关系

| 仓库 | 角色 | 平台如何使用它 |
|------|------|---------------|
| SHUD | 水文求解器 | 在沙箱中编译运行，产生水文数据 |
| rSHUD | 分析工具包 | 读取 SHUD 输出，计算指标和图表 |
| AutoSHUD | 自动化流水线 | 作为可编排脚本链 |
| Zero | 运行时基础 | 提供 Agent/Web/Session 基础设施 |

## 8. 为什么不是只写脚本

| 问题 | 零散脚本 | SHUD-Harness |
|------|---------|-------------|
| 版本追踪 | 依赖人工记录 | 自动锁定代码/数据/环境版本 |
| 数据溯源 | 容易遗漏 | 结构化记录数据来源和预处理 |
| 长任务管理 | 人等命令结束 | 提交后释放，结果回来自动恢复 |
| 失败复盘 | 靠个人经验 | 失败记录和日志结构化保存 |
| 科学报告 | 手工整理 | 自动生成结构化证据报告 |
| 高风险控制 | 靠口头约定 | PI gate 强制审批 |
| 跨仓库检查 | 容易漏测 | 自动兼容性检查 |
| 公式到代码追踪 | 没有 | 公式→推导→代码→验证全链路 |
| 经验沉淀 | 留在个人记忆 | 结构化记录，课题组可复用 |

> 脚本解决"这一次怎么跑"；Harness 解决"怎么让每一次都可追踪、可验证、可复盘"。

## 10. 风险与控制

| 风险 | 后果 | 控制方式 |
|------|------|---------|
| AI 夸大科学结论 | 校准改善被写成"理论已验证" | 语言约束 + PI gate 强制审批 |
| 公式错误被校准掩盖 | 参数补偿隐藏了方程错误 | **搜索后置于理论审查**，高风险变更必须先走证据链 |
| Agent 自行接受推导 | 绕过 PI 审批 | 系统级禁止：Agent 角色不能设置"已接受" |
| 结果不可复现 | 换环境跑不出来 | 版本锁定 + 数据溯源 + 运行记录 |
| 改代码破坏旧输出 | rSHUD / AutoSHUD 断链 | benchmark 对比 + rSHUD roundtrip + 兼容性检查 |
| 平台过度设计 | 建了用不上 | 按阶段验收，Phase 1 只做骨架 |
| 长任务消耗资源 | 浪费推理 token | Park/Resume：等待期间不消耗资源 |

## 11. 拟请老师确认与支持的事项

### 11.1 方向确认

拟请老师确认三项原则：

1. 项目定位为 **PI 主导的科研工程平台**，不做自治科研平台
2. 开发节奏采用 phase gate，按阶段验收，不按自然周硬切
3. 坚持 evidence-first 原则：没有版本记录、数据溯源和运行记录支撑的结论，不进入正式科研报告

### 11.2 资源支持

最小团队配置：

| 角色 | 投入 | 主要职责 |
|------|------|---------|
| PI / 博导 | 低频但关键 | 定方向、审报告、批准高风险决策 |
| 研究生 / 领域研究者 | 高频 | 提出任务、修订计划、跟踪运行、补充诊断 |
| 研究软件工程师 | 主力 | 实现平台、API、UI、runner、验证流程 |

### 11.3 试点选择

建议以 **ccw 暴雨事件** 作为首个真实试点：
- 数据和示例相对成熟
- 运行成本可控
- 可以快速验证运行、图表、报告和兼容性检查
- 适合作为 Phase 2（执行闭环）和 Phase 3（科学运行）的验证场景

## 12. 建成后的价值总结

![[assets/research-loop.png]]
_图 12：从研究问题到可复用科研资产的闭环。_

**对 PI**：更快拿到可判断的证据，而不是零散的日志和"模型好像改善了"的口头汇报。

**对研究生**：减少重复的环境配置、手工运行和结果整理，把时间花在科学问题本身。

**对课题组**：每一次运行沉淀为结构化资产——版本快照、数据溯源、运行记录、证据报告、PI 批注——后人可以接得上、查得到、用得起。

> [!quote] 一句话总结
> 建设 SHUD-Harness 不是追逐 AI 热点，而是解决 SHUD 模型系统进入持续演进阶段后的**工程化科研刚需**：让每一次运行可追踪，每一个改动可验证，每一份报告有证据，每一个结论经 PI 审批。
